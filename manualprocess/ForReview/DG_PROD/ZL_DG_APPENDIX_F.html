☸️ EKS Cluster Optimization

Node Instance Types

Workload TypeRecommended Instance TypesCPU:Memory RatioUse Case
General Purpose m6i.large, m6i.xlarge 1:4, 1:8 Most applications
Compute Optimized c6i.large, c6i.xlarge 1:2, 1:4 CPU-intensive workloads
Memory Optimized r6i.large, r6i.xlarge 1:8, 1:16 Memory-intensive workloads
Storage Optimized i4i.large, i4i.xlarge 1:8, 1:16 High I/O workloads

Cluster Autoscaler Configuration

apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-autoscaler-config
  namespace: kube-system
data:
  # Scale down nodes after 10 minutes of low utilization
  scale-down-delay-after-add: "10m"
  scale-down-delay-after-delete: "10m"
  scale-down-delay-after-failure: "3m"
  scale-down-unneeded-time: "10m"
  scale-down-utilization-threshold: "0.5"
  scan-interval: "10s"
  # Prevent scale-in during business hours
  skip-nodes-with-local-storage: "false"
  skip-nodes-with-system-pods: "true"

🐳 Container Optimization

Resource Limits and Requests

apiVersion: apps/v1
kind: Deployment
metadata:
  name: zl-server-optimized
  namespace: zl-app
spec:
  template:
    spec:
      containers:
      - name: zl-server
        image: .dkr.ecr.us-east-1.amazonaws.com/ue1-zlps-ecr-01:zlserver-1.0.0
        resources:
          requests:
            cpu: "500m"
            memory: "1Gi"
          limits:
            cpu: "2000m"
            memory: "4Gi"
        env:
        - name: JAVA_OPTS
          value: "-Xms512m -Xmx2g -XX:+UseG1GC -XX:MaxGCPauseMillis=200"

Vertical Pod Autoscaler

apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: zl-server-vpa
  namespace: zl-app
spec:
  targetRef:
    apiVersion: "apps/v1"
    kind: Deployment
    name: zl-server
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: '*'
      minAllowed:
        cpu: 100m
        memory: 512Mi
      maxAllowed:
        cpu: 4
        memory: 8Gi
      controlledResources: ["cpu", "memory"]

🐘 Database Performance Tuning

RDS Parameter Groups

# Create custom parameter group
aws rds create-db-parameter-group \
  --db-parameter-group-name zl-postgres-params \
  --db-parameter-group-family postgres14 \
  --description "Optimized parameters for ZL application"

# Set performance parameters
aws rds modify-db-parameter-group \
  --db-parameter-group-name zl-postgres-params \
  --parameters "ParameterName=shared_buffers,ParameterValue={DBInstanceClassMemory/4096*256},ApplyMethod=pending-reboot"

aws rds modify-db-parameter-group \
  --db-parameter-group-name zl-postgres-params \
  --parameters "ParameterName=effective_cache_size,ParameterValue={DBInstanceClassMemory/4096*3/4*1024},ApplyMethod=immediate"

aws rds modify-db-parameter-group \
  --db-parameter-group-name zl-postgres-params \
  --parameters "ParameterName=work_mem,ParameterValue=8388608,ApplyMethod=immediate"

aws rds modify-db-parameter-group \
  --db-parameter-group-name zl-postgres-params \
  --parameters "ParameterName=maintenance_work_mem,ParameterValue=134217728,ApplyMethod=immediate"

⚖️ Load Balancer Optimization

ALB Target Group Settings

# Create optimized target group
aws elbv2 create-target-group \
  --name zl-app-tg \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-12345678 \
  --target-type ip \
  --health-check-path /health \
  --health-check-interval-seconds 30 \
  --health-check-timeout-seconds 5 \
  --healthy-threshold-count 2 \
  --unhealthy-threshold-count 2 \
  --matcher HttpCode=200-399

# Configure ALB attributes for performance
aws elbv2 modify-load-balancer-attributes \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1::loadbalancer/app/zl-alb/1234567890abcdef \
  --attributes Key=idle_timeout.timeout_seconds,Value=60

aws elbv2 set-ip-address-type \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1::loadbalancer/app/zl-alb/1234567890abcdef \
  --ip-address-type dualstack

📊 Horizontal Pod Autoscaling

HPA Configuration

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: zl-server-hpa
  namespace: zl-app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: zl-server
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
      - type: Percent
        value: 100
        periodSeconds: 60
      - type: Pods
        value: 2
        periodSeconds: 60
      selectPolicy: Max

🌐 Network Optimization

Service Mesh Considerations

For high-performance service-to-service communication:

  • Istio: Advanced traffic management, security, and observability
  • Linkerd: Lightweight service mesh with minimal resource overhead
  • AWS App Mesh: Managed service mesh integrated with AWS services
# Istio performance tuning
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: zl-server-dr
  namespace: zl-app
spec:
  host: zl-server.zl-app.svc.cluster.local
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        http1MaxPendingRequests: 10
        http2MaxRequests: 100
        maxRequestsPerConnection: 10
        maxRetries: 3
