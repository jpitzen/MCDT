## Findings Requiring Fix

### F1: `sendSuccess` signature documented incorrectly
**Severity**: HIGH — would cause bugs if followed
**Claim**: `sendSuccess(res, data, 'Created', 201)`
**Actual**: `sendSuccess(res, data, statusCode, message)` — signature is `(res, data = {}, statusCode = 200, message = 'Success')`
**Impact**: Documented call passes string `'Created'` as statusCode and `201` as message — reversed parameter order.
**File**: `backend/src/middleware/errorHandler.js:118`
**Fix**: Change example to `sendSuccess(res, data, 201, 'Created')`

### F2: `emitDeploymentUpdate` signature wrong
**Severity**: MEDIUM — misleading for WebSocket integration work
**Claim**: `websocketServer.emitDeploymentUpdate(deploymentId, { status, phase, progress })`
**Actual**: `emitDeploymentUpdate(deploymentId, eventType, data)` — 3 arguments where `eventType` is a string like `'log'`, `'phase-update'`, `'completed'`
**File**: `backend/src/config/websocketServer.js:442`
**Fix**: Update to show correct 3-arg signature

### F3: WebSocket event names wrong in both hooks
**Severity**: HIGH — would cause silent failures if developer listens for wrong events
**Claim (useWebSocket)**: `deployment:update`, `deployment:completed`
**Actual (useWebSocket)**: `deployment:log`, `deployment:phase-update`, `deployment:progress-update`, `deployment:completed`, `deployment:failed`
**Claim (useDeploymentSocket)**: `terraform:output`, `deployment:warning`, `deployment:rollback`
**Actual (useDeploymentSocket)**: `deployment:terraform-output`, `deployment:error`, `deployment:rollback-completed`
**Files**: `frontend/src/hooks/useWebSocket.js:230-234`, `frontend/src/hooks/useDeploymentSocket.js:138-252`
**Fix**: Correct all event names

### F4: Dual migration system undocumented
**Severity**: MEDIUM — confusing for new developers
**Claim**: "Run via `backend/run-migrations.js`"
**Actual**: `package.json` `db:migrate` uses `sequelize-cli db:migrate`, while `run-migrations.js` runs raw SQL files from `backend/migrations/`. Two separate migration systems coexist.
**Fix**: Document both systems and clarify which is for what

### F5: Debugging endpoint path wrong
**Severity**: LOW — minor path error
**Claim**: `GET /api/logs/deployments/:id`
**Actual**: `GET /api/logs/:deploymentId` — logs route mounted at `/api/logs`, route is `/:deploymentId`
**File**: `backend/src/routes/logs.js:37`, mounted at `server.js:119`
**Fix**: Correct the path

### F6: `styled()` claim is false
**Severity**: LOW — minor inaccuracy
**Claim**: "no `styled()` or CSS modules"
**Actual**: `styled()` is used in `PhaseNavigation.jsx` (3 instances: `CustomConnector`, `PhaseIconContainer`, `StatusBadge`)
**File**: `frontend/src/pages/UnifiedDeploymentWizard/components/PhaseNavigation.jsx:75-115`
**Fix**: Change to "predominantly `sx` prop (rare `styled()` in wizard navigation)"

### F7: Wizard phase names imprecise
**Severity**: LOW — cosmetic but could confuse phase lookups
**Claim**: "credentials → cluster config → database → storage → containers → monitoring"
**Actual**: Phase1Prerequisites → Phase2ClusterConfig → Phase3Database → Phase4Storage → Phase5Deploy → Phase6Operations
**Fix**: Use actual phase component names

### F8: Schema table list incomplete
**Severity**: LOW — omits models that may need consideration
**Claim**: Lists 6 tables
**Actual**: 14 models exist. Missing: `AuditLog`, `AlertChannelConfig`, `ContainerDeployment`, `DatabaseCredential`, `DeploymentSqlScript`, `SharedResource`
**Fix**: Add missing tables or note the list is non-exhaustive

### F9: `credentials.js` applies auth at router level, but not all route files do
**Severity**: LOW — the pattern described is the dominant one
**Claim**: "Auth applied at router level"
**Actual**: Most routes use `router.use(authenticate)` but `auth.js` naturally doesn't (public login/register), and some routes apply per-handler. The pattern is correct as a convention to follow.
**Fix**: No change needed — correct as a convention guide
