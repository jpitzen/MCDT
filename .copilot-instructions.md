This file provides explicit instructions for GitHub Copilot (and similar AI code assistants) when working in this repository.

The primary goal when generating or modifying code in this project is to produce **complete, modular, multi-cloud-capable Terraform configurations** that implement **fully automated deployment** driven entirely by **user-provided variables / configuration files**.

Every generated Terraform codebase **must** strictly follow the phased approach defined in `DeploymentGuide\DeploymentGuide.md`.

## Core Requirements for All Generated Terraform

1. Full automation – zero manual steps after `terraform apply`
   - `terraform init`, `validate`, `plan`, `apply` must be runnable in sequence
   - Include wrapper script (`deploy.sh` / `deploy.ps1`) when appropriate
   - Use remote backend configuration (S3 + DynamoDB, Azure Blob + locking, GCS + locking, etc.)

2. User-driven configuration
   - All important decisions come from variables / tfvars / environment variables
   - Never hard-code credentials, regions, account IDs, resource names, etc.
   - Support multi-cloud via `var.enabled_clouds` list or map

3. Strict adherence to phased approach from DeploymentGuide.md
   Phase 1: Planning and Assessment  
   Phase 2: Configuration and Setup (providers, identity, base networking)  
   Phase 3: Resource Provisioning (compute, storage, databases, etc.)  
   Phase 4: Integration and Automation (CI/CD hooks, monitoring agents, etc.)  
   Phase 5: Testing and Validation  
   Phase 6: Deployment and Rollout (blue-green, canary when applicable)  
   Phase 7: Monitoring and Optimization (post-apply hooks, scaling policies)

   → Use module composition or `depends_on` + naming conventions to enforce phase ordering  
   → Name modules and directories to reflect phases when possible (`modules/phase2-base-networking`, etc.)

4. Multi-cloud best practices
   - Use provider aliases when deploying to multiple regions/accounts simultaneously
   - Abstract common patterns into reusable modules (`modules/vpc`, `modules/subnet-layout`, `modules/vm-group`, etc.)
   - Support at minimum: AWS, Azure, GCP (expandable)

5. Security posture (minimum)
   - Least privilege IAM / RBAC everywhere
   - No public IPs unless explicitly requested via variable
   - Secrets never in Terraform state – use external secret stores + data sources
   - Encrypt everything (EBS, RDS, disks, object storage, transit)
