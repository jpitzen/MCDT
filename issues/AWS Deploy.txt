## IAM Roles & Policies

> **Section Purpose:** Create the IAM roles and policies required for EKS cluster operation, node groups, and add-ons BEFORE creating the cluster. These are foundational prerequisites.

### Required IAM Roles

| Role | Purpose | When Created |
|------|---------|--------------|
| **EKS Cluster Role** | Allows EKS to manage AWS resources | Before cluster creation |
| **EKS Node Role** | Allows EC2 nodes to call AWS APIs | Before node group creation |
| **EBS CSI Driver Role** | Allows EBS volume provisioning | Before installing EBS CSI |
| **EFS CSI Driver Role** | Allows EFS mount operations | Before installing EFS CSI |
| **Cluster Autoscaler Role** | Allows scaling node groups | Before autoscaler deployment |

### Step 1: Create EKS Cluster Role

```bash
# Create trust policy
cat > eks-cluster-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "eks.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

# Create role
aws iam create-role \
  --role-name EKSClusterRole \
  --assume-role-policy-document file://eks-cluster-trust-policy.json

# Attach required policies
aws iam attach-role-policy \
  --role-name EKSClusterRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonEKSClusterPolicy
```

## Security Groups

> **Section Purpose:** Define network security rules before configuring the cluster. With SSM Session Manager, minimal inbound rules are required since SSM uses outbound HTTPS connections.

### Security Group Summary

| Security Group | Purpose | Create When |
|----------------|---------|-------------|
| **EKS Cluster SG** | Control plane communication | Auto-created by EKS |
| **Node SG** | Worker node communication | Before/during node group |
| **SSM Endpoints SG** | VPC endpoints for SSM | Before private cluster |
| **Internal ELB SG** | Internal load balancers | Before deploying services |

### SSM VPC Endpoints Security Group

```bash
# Create security group for SSM VPC endpoints
aws ec2 create-security-group \
  --group-name ssm-endpoints-sg \
  --description "Security group for SSM VPC endpoints" \
  --vpc-id <vpc-id>

# Allow HTTPS from VPC CIDR
aws ec2 authorize-security-group-ingress \
  --group-id <sg-id> \
  --protocol tcp \
  --port 443 \
  --cidr <vpc-cidr>
```

**Inbound:**
- TCP 443 from VPC CIDR (HTTPS for SSM endpoints)

**Outbound:**
- Allow all (default)

> **Note:** When using SSM, you do NOT need to open inbound RDP (3389) or SSH (22) ports.

### Node Security Group

**Inbound:**
- TCP 30000-32767 from VPC CIDR (NodePort range)
- TCP 10250 from control plane SG (kubelet)
- TCP 443 from control plane SG (API server)

**Outbound:**
- Allow all (for pulling images, SSM agent, etc.)

### ELB Security Group

Edit inbound rules to restrict access to VPC CIDR only (internal load balancers).

```bash
  --name <cluster-name>/app/api-keys \
```

## EKS Private Cluster Configuration

> **Section Purpose:** This section covers converting a standard EKS cluster to a fully private configuration for enhanced security. After conversion, the Kubernetes API is only accessible from within the VPC, all load balancers are internal-only, and cluster management uses SSM Session Manager for secure, audited access. This is the recommended setup for production workloads handling sensitive data.

### Overview

Convert a public+private EKS cluster to private-only with internal load balancers accessible via AWS Systems Manager (SSM) Session Manager.

### Architecture Components

| Component | Description |
|-----------|-------------|
| **Public Endpoint** | API accessible over internet (less secure) |
| **Private Endpoint** | API accessible only within VPC (secure) |
| **SSM Session Manager** | Secure shell and port forwarding without bastion (recommended) |
| **Classic ELB (CLB)** | Older load balancer type |
| **ALB/NLB** | Modern load balancer types |
| **Internet-facing LB** | Accessible via public IP |
| **Internal LB** | Accessible only within VPC |

### Key Subnet Tags

- `kubernetes.io/role/elb=1` → Public/internet-facing ELBs
- `kubernetes.io/role/internal-elb=1` → Internal ELBs

---

### Step-by-Step Configuration

> **Subsection Purpose:** This 5-step process walks through converting an EKS cluster to private mode: (1) disable public API endpoint, (2) set up SSM Session Manager for secure access, (3) tag subnets for internal load balancers, (4) configure services with internal LB annotations, (5) verify the configuration.

#### Step 1: Make EKS Control Plane Private

> Disables public internet access to the Kubernetes API. After this change, `kubectl` commands will only work from within the VPC (via SSM port forwarding or from an EC2 instance in the VPC).

```bash
aws eks update-cluster-config \
  --region <region-name> \
  --name <ekscluster-name> \
  --resources-vpc-config endpointPublicAccess=false,endpointPrivateAccess=true
```

**Verify:**

```bash
aws eks describe-cluster \
  --name <ekscluster-name> \
  --region <region-name> \
  --query "cluster.resourcesVpcConfig.{Public:endpointPublicAccess,Private:endpointPrivateAccess}"
```

> **Note:** After this, kubectl only works from inside the VPC (via SSM or EC2 instance).

---

#### Step 2: Configure SSM Session Manager Access (Recommended over Bastion)

> **Why SSM instead of Bastion?**
> - No EC2 bastion cost (~$30+/month saved)
> - No inbound security group rules needed (RDP/SSH ports)
> - No public IP required
> - IAM-based access control
> - Full audit logging in CloudTrail
> - Port forwarding for DB, K8s API, any internal service

**Prerequisites:**

1. AWS CLI v2 with Session Manager plugin installed
2. IAM permissions for SSM

**Install Session Manager Plugin:**

```bash
# Windows (PowerShell as Admin)
Invoke-WebRequest -Uri https://s3.amazonaws.com/session-manager-downloads/plugin/latest/windows/SessionManagerPluginSetup.exe -OutFile SessionManagerPluginSetup.exe
.\SessionManagerPluginSetup.exe

# macOS
curl "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/mac/sessionmanager-bundle.zip" -o "sessionmanager-bundle.zip"
unzip sessionmanager-bundle.zip
sudo ./sessionmanager-bundle/install -i /usr/local/sessionmanagerplugin -b /usr/local/bin/session-manager-plugin

# Linux
curl "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/ubuntu_64bit/session-manager-plugin.deb" -o "session-manager-plugin.deb"
sudo dpkg -i session-manager-plugin.deb
```

**Create VPC Endpoints for SSM (Required for Private Subnets):**

```bash
# Get VPC ID
VPC_ID=$(aws eks describe-cluster --name <ekscluster-name> --region <region-name> --query "cluster.resourcesVpcConfig.vpcId" --output text)

# Get private subnet IDs
SUBNET_IDS=$(aws ec2 describe-subnets --filters "Name=vpc-id,Values=$VPC_ID" "Name=tag:kubernetes.io/role/internal-elb,Values=1" --query "Subnets[].SubnetId" --output text)

# Create security group for VPC endpoints
ENDPOINT_SG=$(aws ec2 create-security-group --group-name ssm-endpoints-sg --description "Security group for SSM VPC endpoints" --vpc-id $VPC_ID --query "GroupId" --output text)

# Allow HTTPS from VPC CIDR
VPC_CIDR=$(aws ec2 describe-vpcs --vpc-ids $VPC_ID --query "Vpcs[0].CidrBlock" --output text)
aws ec2 authorize-security-group-ingress --group-id $ENDPOINT_SG --protocol tcp --port 443 --cidr $VPC_CIDR

# Create VPC endpoints for SSM
for SERVICE in ssm ssmmessages ec2messages; do
  aws ec2 create-vpc-endpoint \
    --vpc-id $VPC_ID \
    --service-name com.amazonaws.<region-name>.${SERVICE} \
    --vpc-endpoint-type Interface \
    --subnet-ids $SUBNET_IDS \
    --security-group-ids $ENDPOINT_SG \
    --private-dns-enabled
done
```

**Port Forwarding to Access Private Resources:**

```bash
# Forward PostgreSQL (pod in private cluster) to localhost:5433
# First, get an EKS node instance ID
INSTANCE_ID=$(aws ec2 describe-instances --filters "Name=tag:eks:cluster-name,Values=<ekscluster-name>" "Name=instance-state-name,Values=running" --query "Reservations[0].Instances[0].InstanceId" --output text)

# Start SSM session with port forwarding
aws ssm start-session \
  --target $INSTANCE_ID \
  --document-name AWS-StartPortForwardingSession \
  --parameters '{"portNumber":["5432"],"localPortNumber":["5433"]}'

# Now connect to PostgreSQL locally
psql -h localhost -p 5433 -U postgres
```

**Execute SQL Scripts via SSM:**

```bash
# Option 1: Port forward to RDS/Pod and run psql locally
aws ssm start-session --target $INSTANCE_ID --document-name AWS-StartPortForwardingSession --parameters '{"portNumber":["5432"],"localPortNumber":["5433"]}'
# In another terminal:
psql -h localhost -p 5433 -U postgres -d mydb -f create_tables.sql

# Option 2: Use kubectl from within the cluster via SSM
aws ssm start-session --target $INSTANCE_ID
# Then inside the session:
kubectl exec -it postgres-pod -- psql -U postgres -d mydb -c "SELECT * FROM users;"
```

> **Legacy Option:** If SSM is not available, see Appendix A for Windows Bastion setup.

#### Step 4: Configure Internal Load Balancer

**Service YAML Example (`zlui-service.yaml`):**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: zlui
  labels:
    app: zlui
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
    # Optional: Force specific subnets
    # service.beta.kubernetes.io/aws-load-balancer-subnets: "subnet-1,subnet-2,subnet-3"
spec:
  type: LoadBalancer
  ports:
    - name: "80"
      port: 80
      protocol: TCP
      targetPort: 80
    - name: "8000"
      port: 8000
      protocol: TCP
      targetPort: 8000
    - name: "fileconnector"
      port: 9975
      protocol: TCP
      targetPort: 9975
    - name: "sharepoint"
      port: 9970
      protocol: TCP
      targetPort: 9970
  selector:
    app: zlui
```

**Deploy Service:**

```bash
kubectl delete svc zlui  # Delete existing service
kubectl apply -f zlui-service.yaml
kubectl get svc zlui -w
```

### Common Issues & Solutions

> **Subsection Purpose:** Quick reference for diagnosing and fixing common problems encountered during private cluster setup. Most issues relate to subnet tagging, load balancer configuration, or routing.

| Issue | Root Cause | Fix |
|-------|------------|-----|
| LB still internet-facing | Wrong subnet tags or missing annotation | Correct tags and redeploy service |
| ELB name too long | AWS Classic ELB limit (32 chars) | Use shorter service names |
| Subnets not recognized | EKS may not detect new tags | Restart EKS controller or reapply service |
| RouteTable misconfigured | Private subnets routing through IGW | Update route tables to use NAT Gateway |

## Storage Setup

> **Section Purpose:** Configures persistent storage backends (EBS and EFS) for Kubernetes workloads, enabling stateful applications and shared file access across pods. Storage must be configured before deploying stateful applications.

### EBS for Stateful Applications (Zookeeper)

> **Subsection Purpose:** Configures Amazon Elastic Block Store (EBS) as persistent block storage for single-pod stateful workloads like Zookeeper that need fast, persistent disk with `ReadWriteOnce` access mode.

**Purpose:** Provides block storage for stateful applications like Zookeeper.

#### 1. Create IAM Role for EBS CSI Driver

```bash
eksctl create iamserviceaccount \
  --name ebs-csi-controller-sa \
  --namespace kube-system \
  --cluster $CLUSTER_NAME \
  --attach-policy-arn arn:aws:iam::$AWS_ACCOUNT_ID:policy/AmazonEBSCSIDriverPolicy \
  --approve \
  --role-name AmazonEKS_EBS_CSI_DriverRole
```

**Single-line version:**

```bash
eksctl create iamserviceaccount --name ebs-csi-controller-sa --namespace kube-system --cluster $CLUSTER_NAME --attach-policy-arn arn:aws:iam::$AWS_ACCOUNT_ID:policy/AmazonEBSCSIDriverPolicy --approve --role-name AmazonEKS_EBS_CSI_DriverRole
```

#### 2. Install EBS CSI Driver Add-On

```bash
aws eks create-addon \
  --cluster-name $CLUSTER_NAME \
  --addon-name aws-ebs-csi-driver \
  --service-account-role-arn arn:aws:iam::$AWS_ACCOUNT_ID:role/AmazonEKS_EBS_CSI_DriverRole \
  --region $REGION
```

#### 3. Create StorageClass (`ebs-sc.yaml`)

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-sc
provisioner: ebs.csi.aws.com
volumeBindingMode: WaitForFirstConsumer
```

```bash
kubectl apply -f ebs-sc.yaml
```

#### 4. Create PersistentVolumeClaim (`zk-pvc.yaml`)

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: zk-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: ebs-sc
  resources:
    requests:
      storage: 20Gi
```

```bash
kubectl apply -f zk-pvc.yaml
kubectl get pvc
kubectl describe pvc zk-pvc
```

#### 5. Deploy Zookeeper StatefulSet

```bash
kubectl apply -f zlzookeeper.yaml
kubectl get statefulsets
kubectl describe pod zlzookeeper-0
```

#### 6. Verification


### EFS for Shared Workloads

> **Subsection Purpose:** Configures Amazon Elastic File System (EFS) for shared network storage. Unlike EBS, EFS supports `ReadWriteMany` access mode, allowing multiple pods to read and write to the same file system simultaneously. Ideal for shared config files, uploads, or any data that needs to be accessed by multiple replicas.

**Purpose:** Provides shared, network-mounted storage for multiple pods.

#### 1. Create EFS File System

```bash
aws efs create-file-system --region <region> --performance-mode generalPurpose
```

#### 2. Create Access Point

```bash
aws efs create-access-point \
  --file-system-id <fs-id> \
  --posix-user Uid=1000,Gid=1000 \
  --root-directory '{"Path":"/","CreationInfo":{"OwnerUid":1000,"OwnerGid":1000,"Permissions":"755"}}'
```

#### 3. Install EFS CSI Driver

```bash
helm repo add aws-efs-csi-driver https://kubernetes-sigs.github.io/aws-efs-csi-driver/
helm repo update

helm install aws-efs-csi-driver aws-efs-csi-driver/aws-efs-csi-driver \
  --namespace kube-system \
  --set controller.serviceAccount.create=true \
  --set controller.serviceAccount.name=efs-csi-controller-sa
```

#### 4. Create StorageClass (`efs-sc.yaml`)

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: efs-sc
provisioner: efs.csi.aws.com
```

```bash
kubectl apply -f efs-sc.yaml
```

#### 5. Create PVC (`efs-pvc.yaml`)

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: efs-pvc
spec:
  accessModes:
    - ReadWriteMany
  storageClassName: efs-sc
  resources:
    requests:
      storage: 5Gi
```

```bash
kubectl apply -f efs-pvc.yaml
kubectl get pvc
kubectl describe pvc efs-pvc
```

## Application Deployment

> **Section Purpose:** Deploy applications to the configured cluster. This phase assumes storage and cluster configuration are complete.

### Deploy Application with EFS

Example deployment using EFS shared storage:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: <aws_account_id>.dkr.ecr.<region>.amazonaws.com/my-app-repo:latest
        ports:
        - containerPort: 80
        volumeMounts:
        - name: efs-storage
          mountPath: /mnt/data
      volumes:
      - name: efs-storage
        persistentVolumeClaim:
          claimName: efs-pvc
```

## Cluster Autoscaling

> **Section Purpose:** Enables automatic horizontal scaling of EKS worker nodes. When pods are pending due to insufficient resources, the Cluster Autoscaler provisions new EC2 instances. When nodes are underutilized, it scales down to reduce costs. Requires IAM permissions to modify Auto Scaling Groups.

**Purpose:** Automatically adjusts node counts based on workload demand.

| Component | Description |
|-----------|-------------|
| Cluster Autoscaler | Monitors pending pods and scales node groups |
| Node Groups/ASGs | EC2 instances forming worker nodes |
| IAM Role & Policy | Grants permissions to modify AWS resources |
| Scaling Limits | Defines min/max nodes per group |

### Setup Commands

```bash
# Create IAM policy
aws iam create-policy --policy-name AmazonEKSClusterAutoscalerPolicy ...

# Create IAM service account
eksctl create iamserviceaccount --name cluster-autoscaler ...

# Update node group configuration
aws eks update-nodegroup-config ...

# Deploy autoscaler
kubectl apply -f cluster-autoscaler.yaml

# Validate
kubectl logs -n kube-system -f deployment/cluster-autoscaler
```

### Troubleshooting Workflow

> **Subsection Purpose:** A systematic 7-step approach to diagnosing cluster issues. Start with pod status, drill down into logs and events, then expand to node health, resources, network, and configuration. Follow this order to efficiently identify root causes.

#### Step 1: Check Pod Status

```bash
kubectl get pods --namespace=<namespace>
```

#### Step 2: Describe Problem Pod

```bash
kubectl describe pod <pod-name>
```

Check the **Events** section for errors.

#### Step 3: View Pod Logs

```bash
kubectl logs -f <pod-name>
```

#### Step 4: Check Node Health

```bash
kubectl get nodes
kubectl describe node <node-name>
```

Look for `NotReady`, `DiskPressure`, or resource exhaustion.

#### Step 5: Check Resource Quotas

```bash
kubectl get resourcequotas -A
kubectl top nodes
kubectl top pods -A
```

#### Step 6: Verify Network Connectivity

```bash
kubectl exec -it <pod-name> -- /bin/bash
ping <service-name>
curl <service-name>:<port>
```

#### Step 7: Review Configuration

```bash
kubectl get configmaps
kubectl get secrets
kubectl describe configmap <name>
kubectl describe secret <name>
```

## Best Practices

1. **Always use internal load balancers** for private clusters
2. **Tag subnets correctly** to control ELB placement
3. **Use metrics-server** for resource monitoring
4. **Enable logging** for troubleshooting
5. **Use SSM Session Manager** instead of bastion hosts (cost-effective, IAM-based, audited)
6. **Create VPC endpoints** for SSM in private subnets
7. **Regularly check events** for cluster issues
8. **Monitor resource usage** to prevent exhaustion
9. **Verify configurations** before deploying
10. **Use namespaces** to organize resources
11. **Keep IAM roles** scoped to minimum permissions
12. **Use SSM port forwarding** to access databases and internal services securely
