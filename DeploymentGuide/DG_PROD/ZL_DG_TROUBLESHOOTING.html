Troubleshooting Guide

Reference Updated: Dec 23, 2025

This troubleshooting guide provides solutions for common issues encountered during ZL application deployment on AWS EKS.

🔍 Common Issues & Solutions

  • EKS cluster connectivity problems
  • CSI driver installation failures
  • Application deployment issues
  • Network and ingress configuration
  • Database connectivity problems
  • Security and permissions issues

🚨 EKS Cluster Issues

Cluster Not Accessible

Problem: Cannot connect to EKS cluster after creation
Symptoms: kubectl commands fail with connection errors
Solution:
  1. Verify AWS credentials: aws sts get-caller-identity
  2. Update kubeconfig: aws eks update-kubeconfig --name your-cluster-name --region us-east-1
  3. Check cluster status: aws eks describe-cluster --name your-cluster-name --region us-east-1
  4. Verify IAM permissions for EKS access

Nodes Not Joining Cluster

Problem: EC2 instances created but not appearing in kubectl get nodes
Symptoms: Cluster shows 0 nodes
Solution:
  1. Check node group status: aws eks describe-nodegroup --cluster-name your-cluster-name --nodegroup-name your-nodegroup --region us-east-1
  2. Verify VPC configuration and security groups
  3. Check CloudFormation stack events for node group
  4. Review IAM roles for node instances

💾 CSI Driver Issues

EBS Volumes Not Attaching

Problem: PersistentVolumeClaims remain in Pending state
Symptoms: PVC events show "waiting for volume to be created"
Solution:
  1. Verify EBS CSI driver installation: kubectl get pods -n kube-system | grep ebs
  2. Check IAM permissions for EBS access
  3. Verify storage class: kubectl get storageclass
  4. Check PVC events: kubectl describe pvc your-pvc-name

EFS Mount Issues

Problem: Pods cannot mount EFS volumes
Symptoms: Mount errors in pod logs
Solution:
  1. Verify EFS CSI driver: kubectl get pods -n kube-system | grep efs
  2. Check EFS security groups allow NFS traffic
  3. Verify VPC configuration for EFS mount targets
  4. Check IAM permissions for EFS access

🌐 Network & Ingress Issues

ALB Not Created

Problem: Ingress resources don't create Application Load Balancers
Symptoms: Ingress shows ADDRESS as <pending>
Solution:
  1. Verify ALB controller installation: kubectl get deployment -n kube-system aws-load-balancer-controller
  2. Check IAM permissions for ALB controller
  3. Verify ingress class annotation
  4. Check CloudFormation stack for ALB controller

Service Not Accessible

Problem: Application not reachable via ALB DNS
Symptoms: Connection timeouts or 404 errors
Solution:
  1. Check service endpoints: kubectl get endpoints
  2. Verify pod health: kubectl get pods
  3. Check ingress rules and paths
  4. Verify security groups allow traffic

🗄️ Database Issues

Database Connection Failures

Problem: Application cannot connect to RDS PostgreSQL
Symptoms: Connection timeout or authentication errors
Solution:
  1. Verify RDS endpoint and port
  2. Check security groups allow database traffic
  3. Verify database credentials in secrets
  4. Test connection from bastion host

🔒 Security & Permissions

IAM Permission Errors

Problem: AWS API calls fail with AccessDenied
Symptoms: Operations fail with IAM errors
Solution:
  1. Check IAM user/role policies
  2. Verify correct region and account
  3. Check for policy conditions or restrictions
  4. Use IAM policy simulator to test permissions

ECR Pull Errors

Problem: Pods cannot pull images from ECR
Symptoms: ImagePullBackOff status
Solution:
  1. Verify ECR repository exists and is accessible
  2. Check IAM permissions for ECR access
  3. Verify image tags and digests
  4. Check VPC endpoints for ECR

🦓 ZL Application Issues

These issues are specific to the ZL application deployment and were discovered during deployment testing.

🚨 CRITICAL: ZOO_SERVERS Corrupts zkQuorum.cfg

Problem: ZLServer fails to connect to ZooKeeper cluster
Symptoms: "Unable to resolve address: zlzookeeper" or "ZLZKAuth is not initialized"
Root Cause: The ZOO_SERVERS environment variable triggers sed commands in zlserver-startup.sh that corrupt the zkQuorum.cfg file
Solution:
  1. REMOVE the ZOO_SERVERS key from zlapp-config ConfigMap
  2. Provide ZooKeeper configuration via zkclient-config ConfigMap with proper zkQuorum.cfg
  3. Use the aws/zlapp-config.yaml template which does NOT include ZOO_SERVERS
# CORRECT zlapp-config.yaml - NO ZOO_SERVERS
apiVersion: v1
kind: ConfigMap
metadata:
  name: zlapp-config
data:
  # ZOO_SERVERS: DO NOT ADD THIS - it corrupts zkQuorum.cfg
  TIKA_HOST: zltika
  TIKA_PORT: "9972"

zkQuorum.cfg Requires Full DNS Names

Problem: ZLServer cannot reach ZooKeeper pods
Symptoms: "Unable to resolve address: zlzookeeper-0" or connection timeouts
Root Cause: Short hostnames (zlzookeeper-0) are used instead of full Kubernetes DNS names
Solution:
  1. Use full DNS names in zkQuorum.cfg: zlzookeeper-0.zk-hs.default.svc.cluster.local
  2. The headless service (zk-hs) must exist for DNS resolution to work
  3. Verify with: kubectl run test --rm -it --image=busybox -- nslookup zlzookeeper-0.zk-hs.default.svc.cluster.local
# CORRECT zkQuorum.cfg format
zkQuorum.0=0~~zlzookeeper-0.zk-hs.default.svc.cluster.local~~2181~~2888~~3888~~/var/ZipLip/DATA/ZooKeeper/zk0
zkQuorum.1=1~~zlzookeeper-1.zk-hs.default.svc.cluster.local~~2181~~2888~~3888~~/var/ZipLip/DATA/ZooKeeper/zk1
zkQuorum.2=2~~zlzookeeper-2.zk-hs.default.svc.cluster.local~~2181~~2888~~3888~~/var/ZipLip/DATA/ZooKeeper/zk2

ConfigMap Files Are Read-Only

Problem: Startup script fails with "sed: cannot rename ... Device or resource busy"
Symptoms: ZLServer pod crashes during startup, sed errors in logs
Root Cause: Kubernetes mounts ConfigMap volumes as read-only; startup scripts try to modify files in-place
Solution:
  1. Mount ConfigMap files to a templates directory (e.g., /opt/ZipLip/config-templates/)
  2. Copy files to writable locations in the container startup command
  3. See the aws/zlserver-deployment.yaml for the correct pattern
# Startup command must copy config files
args:
  - |
    # Copy from read-only ConfigMap to writable location
    cp /opt/ZipLip/config-templates/tcdb.cfg /opt/ZipLip/bin/zk/tcdb.cfg
    cp /opt/ZipLip/config-templates/zkQuorum.cfg /opt/ZipLip/ZLZooKeeper/config/zkQuorum.cfg
    /opt/ZipLip/bin/zlserver-startup.sh

Database SSL Certificate Error

Problem: Database connection fails with SSL certificate error
Symptoms: "The driver could not establish a secure connection... trustServerCertificate property is set to false"
Root Cause: JDBC connection requires trusted certificate or explicit trust configuration
Solution:
  1. For development/self-signed certs: Add TrustServerCertificate=true to JDBC URL
  2. For production: Use proper SSL certificates from a trusted CA
  3. For AWS RDS: Usually not needed as RDS uses trusted certificates
# tcdb.cfg with TrustServerCertificate (for self-signed certs only)
_db.param.jdbcUrl=jdbc:sqlserver://your-host:1433;TrustServerCertificate=true
_db.param.DatabaseName=ZLDB

Tika Connection Failed

Problem: Document conversion fails during ZLServer Phase3 initialization
Symptoms: "Thread has already exited; Possibly unable to connect!" in BatchRemoteInvoker
Root Cause: Tika service not running or environment variables not configured correctly
Solution:
  1. Verify Tika deployment is running: kubectl get pods -l app=zltika
  2. Verify Tika service exists: kubectl get svc zltika
  3. Verify zlapp-config contains Tika environment variables
  4. Ensure zltika deployment is running BEFORE zlserver starts
# Verify Tika is running
kubectl get pods -l app=zltika
kubectl get svc zltika

# Verify environment variables
kubectl get configmap zlapp-config -o yaml | Select-String -Pattern "TIKA|tika"
# Expected output should include:
#   TIKA_HOST: zltika
#   TIKA_PORT: "9972"
#   __tika.ZLTikaService.Host: zltika
#   __tika.ZLTikaService.Port: "9972"

🔴 CRITICAL: Tika Container Exits Before Listener Starts

Problem: Tika container exits immediately after starting
Symptoms:
  • Pod shows Running but Tika not listening on port 9972
  • zlserver Phase3 fails with "Connection refused" to Tika
  • netstat -tlnp | grep 9972 shows nothing

Root Cause: The zltikadiag.sh script runs the Java process in background and shell exits before the listener binds to the port
🔴 CRITICAL FIX: Override the container command with sleep infinity 
# REQUIRED command override in zltika deployment
containers:
- name: zltika
  image: .dkr.ecr.us-east-1.amazonaws.com/ue1-zlps-ecr-01:zltika11.1.0-b1140
  # This command is CRITICAL - without sleep infinity, the container exits
  command:
    - /bin/sh
    - -c
    - cd /opt/ZipLip/ZLTikaConvertor/bin && ./zltikadiag.sh ZLTika /tmp/ZLTika & sleep infinity
  ports:
  - containerPort: 9972

Explanation: The & sleep infinity runs the Tika script in background and keeps the shell alive, allowing the Java process time to bind to port 9972.

Verify Fix:

# Check Tika is listening
kubectl exec $(kubectl get pods -l app=zltika -o name) -- netstat -tlnp | grep 9972
# Expected: tcp 0 0 0.0.0.0:9972 0.0.0.0:* LISTEN

Superadmin Login Failing - AliasLookup NullPointerException

Problem: Cannot login to ZLUI with superadmin credentials
Symptoms:
  • Login page loads but authentication fails
  • Logs show: java.lang.NullPointerException: Cannot invoke "com.zlti.pm.acct.AliasLookup.findAlias..." because the return value of "AliasLookup.getInstance()" is null

Root Cause: ZLUI pod started before ZLServer finished initializing, aliasSync not completed
Solution: Restart the ZLUI deployment after ZLServer shows "Done initializing" 
# First verify zlserver is initialized
kubectl logs deployment/zlserver | grep -i "done init"
# Should show: Done initalizing ZLServer

# Then restart zlui
kubectl rollout restart deployment zlui

# Wait for new pod
kubectl get pods -l app=zlui -w

# Verify aliasSync completed
kubectl logs deployment/zlui | grep -i "alias"
# Should show: Creating New Version pm_alias_...

Prevention: In Phase 8, always wait for ZLServer to show "Done initializing" in logs before deploying ZLUI.

PVC Not Found - Name Mismatch

Problem: Pods stuck in Pending state
Symptoms: Events show "persistentvolumeclaim not found"
Root Cause: PVC names in deployment don't match actual PVC resources
Solution:
  1. For AWS EFS: Use -efs suffix (e.g., zlservertemp-efs)
  2. Verify PVC exists: kubectl get pvc
  3. Ensure deployment references match exactly
# List all PVCs and their status
kubectl get pvc
# Check deployment PVC references
kubectl get deployment zlserver -o yaml | grep claimName

✅ ZLServer Initialization Verification

Verifying Successful Deployment

After deployment, verify that ZLServer has fully initialized:

  1. All pods should be in Running state
  2. ZLServer logs should show "Done initalizing ZLServer"
  3. No exceptions should appear during the initialization phases
# Check all pods
kubectl get pods

# Verify ZLServer initialized successfully
kubectl logs deployment/zlserver | grep -i "done init"
# Expected: "Done initalizing ZLServer"

# Check for errors
kubectl logs deployment/zlserver | grep -i "exception\|error\|failed"
